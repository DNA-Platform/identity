Identified code review priorities across TypeScript vector implementation
Identified code review priorities across TypeScript vector implementation
Good to be looking at this with you. The config files (tsconfig, vite, package.json) are standard scaffolding and look fine, so I'll focus on vector.ts, which is where the actual logic lives. One caveat up front: I can't see ./array, so anything that depends on how A behaves (its constructor signature, indexing, out-of-bounds semantics) I'm inferring rather than verifying.
A few of these are real bugs, not just style.
The constructor does nothing meaningful with its branch.
￼
ts
super(data instanceof A ? data : data)
Both arms of the ternary are identical, so this is just super(data) with extra steps. I suspect the intent was to convert a plain Position[] into an A<Position> when it isn't one already — but as written, a raw array is passed straight through. If A's constructor only accepts A<Position>, this is a latent bug; if it accepts both, the check is dead code. Either way it wants cleaning up. The same redundant data instanceof A ? data : data appears again in the v helper, so whatever the fix is, it's needed in both places.
norm initializes the max-tracker wrong.
￼
ts
let maxVal = f32.MIN_VALUE;
In AssemblyScript f32.MIN_VALUE is the smallest positive value (~1.4e-45), following the JS convention — not the most-negative value. So for any dataset whose x-components are all negative, maxVal never updates and your range is garbage. You want -f32.MAX_VALUE (or f32.NEGATIVE_INFINITY). The minVal = f32.MAX_VALUE line is correct, which makes this asymmetry easy to miss.
norm also normalizes y using x's statistics.
You compute minVal/range from this[i][0] only, then apply that same minVal and range to the y-component:
￼
ts
min + ((p[1] - minVal) / range) * targetRange
If x and y share a coordinate space that's intentional, but if they have different scales it'll distort y. Worth a comment confirming which you mean. And there's no guard for range === 0 (all x equal), which gives you NaN/Inf — same issue in linspace when count === 1((max - min) / (count - 1) divides by zero).
plus/minus assume equal length. No check that other.length === this.length, and other[i] past the end is an out-of-bounds read — which under WASM either traps or returns junk depending on A. A length guard (or at least an assert) would save a confusing crash later.
magnitude returns this on the non-complex path. Every other operation returns a fresh Vector, but here a real vector returns itself by reference, so callers can accidentally alias and mutate the original. Returning a copy keeps the value semantics consistent. Separately, Math.sqrt(...) as f32 rounds an f64 result down — Mathf.sqrt does the work in f32 directly, which is cheaper and avoids the double-rounding.
Smaller things: f is imported but never used (noUnusedLocals will flag it under tsc, and asc may warn); and times uses this.select(...) while plus/minus hand-roll a loop with new A<Position>() + push — both work, but picking one style makes the class easier to read.
Want me to write up corrected versions of norm and the constructor, or would you rather I hold off until you've confirmed the x/y normalization question? That one changes what the fix should be.
