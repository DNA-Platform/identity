Dissected Electron CDP architecture and MSIX constraints thoroughly
Short version: the port really is gated at process initialization, and nothing in stock Chromium/Electron lets you turn the browser-level CDP server on after the fact. But the two protocols you're conflating behave differently, and one of them gives you a live-process path. Let me separate them.
The browser CDP endpoint (what connectOverCDP needs)
Playwright's connectOverCDP talks to the Chromium DevToolsHttpHandler — the HTTP server at http://host:port/json/version that hands back the browser webSocketDebuggerUrl. That handler is constructed during browser-process startup if and only if --remote-debugging-port (or --remote-debugging-pipe) is present in the command line. There is no runtime API — no app.enableRemoteDebugging(), no IPC, no documented switch — to start it later. So an untouched, already-running process simply has no such server, and there's no clean way to make one appear.
Environment variables don't help here. Chromium has no general CHROME_REMOTE_DEBUGGING_PORT-style env var that injects the switch; the only env vars Electron honors broadly are things like ELECTRON_RUN_AS_NODE, which is irrelevant to this. Unless the app's own code reads an env var and forwards it as a flag (some do — worth grepping the app's main bundle), env alone is a dead end.
The realistic no-source-change method: relaunch with package identity
The thing that actually works for an MSIX app without touching its source is to relaunch it with the flag while preserving package identity. You can't just exec the inner exe out of C:\Program Files\WindowsApps\… — it'll usually run without identity and break storage/state. And shell activation (shell:AppsFolder\<PFN>!<AppId>) doesn't reliably forward arbitrary args. The documented tool is the Appx module cmdlet:
powershell
Invoke-CommandInDesktopPackage -PackageFamilyName "<PFN>"
`
-AppId "<AppId>" -Command "App.exe" -Args "--remote-debugging-port=9222"
This runs the package's exe inside its container with identity intact and passes the switch through. For most MSIX Electron apps the logged-in/session state lives on disk in the package's local data, so a relaunch typically restores it — you usually don't lose much by killing and reopening with the flag. This is the answer I'd actually reach for.
--inspect and the Node main process — a different protocol
--inspect is not the same surface. It opens the V8 Inspector (default :9229) scoped to the main process's V8 isolate — JS debugging of your main process, speaking the Inspector protocol. It does not expose page/target CDP, so connectOverCDP can't drive pages through it. Two genuinely different endpoints.
Does it survive MSIX activation? The flag itself isn't stripped by MSIX — the obstacle is identical to --remote-debugging-port: shell activation won't deliver the arg, so you need the same Invoke-CommandInDesktopPackage (or a manifest-declared launch) to get it onto the command line. Once it reaches the process, Electron's main process honors --inspect/--inspect-brk normally.
The one place --inspect becomes interesting for your goal: you can enable the V8 inspector on an already-running Node process via process._debugProcess(pid) from a separate Node instance. On Windows this is implemented through a remote-thread mechanism rather than a signal, and it does work against Node processes. Against an Electron main process it's hit-or-miss (depends on the embedded debug-signal handler being registered), so treat it as opportunistic, not reliable. If it attaches, you then have JS execution in the main process, and from there you can script CDP against any webContents via Electron's built-in webContents.debugger.attach() / sendCommand bridge. That gets you CDP-equivalent control of a page on a live process — but you're driving it by hand through main-process JS, not via connectOverCDP.
Native injection
Yes, in principle you could inject a DLL or attach WinDbg and call into Chromium to construct the DevToolsHttpHandler on the right thread. In practice this is a reverse-engineering project: content is statically linked, the start path isn't an exported API, it has strict thread-affinity requirements (UI/IO sequencing), and it's recompiled-version-specific. Not a method, more a research effort, and not something I'd build tooling on.
So, ranked by what I'd actually do: relaunch with --remote-debugging-port via Invoke-CommandInDesktopPackage (clean, no source change, gets you real connectOverCDP); fall back to _debugProcess → webContents.debugger only if a relaunch is genuinely unacceptable and you can tolerate the fragility; skip injection.
One thing worth checking before any of this: does the app even allow --remote-debugging-port? Some hardened Electron apps strip or refuse it in their own argument handling. Quick test — relaunch with the flag and curl http://127.0.0.1:9222/json/version; if you get a JSON blob with a webSocketDebuggerUrl, you're done.
