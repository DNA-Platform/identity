Synthesized architectural tradeoffs between two IPC frameworks
The short version: gRPC is contract-first, strongly typed, and language-agnostic but fights you on Windows transport; StreamJsonRpc is schema-less, dynamic, and built for exactly the kind of local Windows IPC you're describing but gives you weaker type guarantees across the boundary.
Here's how they actually differ on the axes that matter for Node ↔ .NET.
Transport, and the Windows named-pipe wrinkle. This is the decisive issue for your scenario. StreamJsonRpc is transport-agnostic — it runs over any Stream, so named pipes, stdio, or a socket all work, and named pipes are the natural, low-overhead choice on Windows. gRPC is built on HTTP/2, which historically meant a localhost TCP port. Both stacks have added named-pipe support (Kestrel got a named-pipe transport, and Grpc.Net.Client can connect through a SocketsHttpHandler.ConnectCallback wrapping a NamedPipeClientStream), but the friction is on the Node side: @grpc/grpc-js has solid Unix-domain-socket support but poor-to-nonexistent Windows named-pipe support. So in practice a gRPC setup tends to land you on a loopback TCP port — meaning a port to allocate, possible firewall prompts, and a surface that's technically network-bound even if you bind to 127.0.0.1. (I'd treat the exact named-pipe support level in the current grpc-js and .NET releases as worth verifying before you commit, since it's evolved over the last few versions.)
Node interop quality. gRPC has first-class clients on both sides via .proto codegen — that's its biggest strength. StreamJsonRpc has no official Node client, but it pairs cleanly with vscode-jsonrpc on the Node side, because StreamJsonRpc's header-delimited message framing is the same LSP-style framing that library uses (both descend from Microsoft's LSP lineage). So "no official client" is less scary than it sounds — you're using a mature, compatible JSON-RPC 2.0 implementation, just matching the framing and encoding.
Contracts and type safety. gRPC wins clearly. You define messages in .proto, get generated types on both sides, and get disciplined schema-evolution rules (field numbering, backward compatibility). StreamJsonRpc is dynamic — the contract is implicit in your method signatures, with no codegen and no enforced cross-boundary schema. Flexible to iterate on, but you can drift a parameter shape on one side and not find out until runtime.
Serialization and performance. gRPC uses binary Protobuf — compact and fast, and it pulls ahead on high-throughput or large payloads. StreamJsonRpc defaults to JSON but supports MessagePack, which closes most of the gap. For typical local IPC volumes the difference is rarely the bottleneck; the named-pipe-vs-TCP and setup-complexity questions usually matter more.
Bidirectional communication. Both support it, but differently. StreamJsonRpc is genuinely peer-to-peer: either side can invoke methods on the other, register callbacks, send notifications, and it has built-in cancellation and progress reporting — very natural if your two processes are equals calling each other. gRPC's bidirectional streaming is a single typed stream of messages over one call, which is great for streaming data but more awkward if what you want is arbitrary RPC in both directions.
Setup cost. StreamJsonRpc is far less ceremony to stand up for a 1:1 local link. gRPC brings codegen, build tooling, and more configuration — which pays off in larger or polyglot systems, and is overkill for a single Node helper talking to a single .NET host.
So a rough decision rule: if this is a local, Windows-first, peer-to-peer link between one Node process and one .NET process and you value low-friction named-pipe IPC and bidirectional calls, StreamJsonRpc (with vscode-jsonrpc on Node) is the better-fitting tool. If you anticipate strict typed contracts, schema versioning across a larger surface, high-throughput streaming, or eventually more languages/services, gRPC's discipline is worth the transport headaches — and you'd likely just accept a loopback TCP port rather than chase named pipes.
What does the workload look like — mostly request/response, or heavy streaming — and is this strictly the two processes on one machine?
