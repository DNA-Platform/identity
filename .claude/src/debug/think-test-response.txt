Synthesized Windows automation frameworks, maturity levels, and practical implementation paths for AI assistants
Here's the current landscape, organized around your four questions. Short version up front: beyond MCP the action is in three places — vision-based "computer use" model APIs, Windows-native accessibility automation (which is more useful than people expect), and a thin layer of agent-orchestration protocols/SDKs sitting above MCP. For a coding assistantspecifically, the highest-leverage move is usually not a new protocol but wrapping deterministic UIA-based tooling as MCP servers or function-call tools.
1. Letting an AI agent control desktop applications
Computer-use model APIs are the most general approach and are genuinely available today. Anthropic introduced computer use in October 2024 as an API capability you run yourself, typically in a sandboxed VM, and it supports full desktop control including terminal commands, native applications, and file system operations — not just a browser. OpenAI shipped Operator in January 2025 (the CUA model) and folded it into ChatGPT Agent in July 2025; Google has a Gemini-based equivalent. The mechanism is the same across all of them: screenshot the screen, identify fields and buttons from pixels, calculate coordinates, move a virtual mouse and type.
Lapu AI + 3
The honest maturity assessment: this works on anything a human can see, including legacy and Citrix apps with no API, but it's computationally expensive, slower due to the screenshot loop, and prone to hallucinating UI elements on cluttered interfaces. Production deployments are still mostly supervised. For Anthropic, the current path uses the computer-use-2025-11-24beta header with Claude 4.x models — check docs.claude.com for the exact current models and tool spec rather than trusting a blog.
ModelGate AI
UFO / UFO² (Microsoft Research) is the most mature Windows-specific desktop agent and is open source: https://github.com/microsoft/UFO. The original UFO was the first UI agent specifically tailored for task completion on Windows OS (arXiv 2402.07939). The 2025 rewrite, UFO² ("The Desktop AgentOS," arXiv 2504.14603), is the interesting one — it explicitly rejects pure screenshot interaction. A hybrid control detection pipeline fuses Windows UI Automation (UIA) with vision-based parsing, with speculative multi-action planning to reduce per-step LLM overhead. Architecturally it's a HostAgent that parses the goal and launches apps, plus one AppAgent per application running a ReAct loop with hybrid control detection and a Puppeteer executor that chooses between GUI actions and native APIs. Still research-grade (Microsoft labels it "not an official Windows feature"), but it's the best blueprint available for how a Windows agent should be built, and you can run it.

2. Windows-specific automation (UIA, Win32, accessibility)
This is the part most people underweight, and for a coding assistant it's probably the most practical. These are deterministic, object-based libraries your assistant can drive by writing code — far more reliable than pixel-clicking:
• pywinauto (Python) — drives native Windows apps via UIA and Win32 backends; object-level, not image recognition. The most-used option in the Python ecosystem and actively maintained.
• FlaUI (.NET/C#) — wraps Microsoft UI Automation (UIA2/UIA3); the recommended modern starting point alongside pywinauto, both actively maintained with solid object-based locators for Win32/WinForms/WPF.
Test Guild
• Python-UIAutomation-for-Windows — direct UIA bindings if you want lower-level control in Python.
• WinAppDriver — WebDriver/Appium-style. Worth a maturity warning: Appium's Windows Driver docs now point to NovaWindows Driver as the modern replacement; if WinAppDriver works keep it, but don't start new projects on it. Winium is effectively abandoned.
Test Guild
For visual/Citrix/game cases where object access isn't available, the fallback is CV-based tools — SikuliX, Airtest, or AskUI's vision agent.
The synthesis worth internalizing: UFO²'s whole design thesis is that UIA gives you reliable, inspectable structure and vision fills the gaps. So the strongest setup for an AI coding assistant is to expose FlaUI/pywinauto operations as callable tools and reserve computer-use vision for when the accessibility tree comes up empty.
3 & 4. Protocols beyond MCP, and AI-to-application standards
Windows 11 native MCP + App Actions is the most directly relevant emerging standard for you. Announced at Build 2025, it's now actually shipping in preview: native MCP support went to Windows Insiders in build 26220.7344, with two built-in connectors (File Explorer and Windows Settings) and an on-device registry (ODR) that sandboxes connectors with identity and audit trails. App Actions lets developers expose specific app features that agents can invoke, available in private developer preview with select partners. Maturity: real but early/preview, and Microsoft is leading heavily with security framing (cross-prompt injection, tool poisoning, agent identity via Entra). Developer hub: https://developer.microsoft.com/en-us/windows/agentic.
Windows Forum
Windows Developer Blog
A2A (Agent2Agent) solves a different axis than MCP — agent-to-agent rather than agent-to-tool. Google announced it April 2025; it uses HTTP, Server-Sent Events, and JSON-RPC 2.0, with Agent Cards at /.well-known/agent-card.json for capability discovery, and sits alongside MCP rather than replacing it. It's now governed by the Linux Foundation under Apache-2.0.Maturity: still relatively new with limited production evidence as of early 2026, but broad enterprise backing (150+ orgs). Site: https://a2a-protocol.org. Probably overkill unless you're orchestrating multiple independent agents; for a single coding assistant it adds little over MCP.

Orchestration / structured-tool-use SDKs (the layer above raw function calling):
• Microsoft Agent Framework — the consolidation play. Version 1.0 merges Semantic Kernel and AutoGen into one SDK (Python + .NET) with built-in MCP client support and A2A, plus graph-based workflows. Most natural fit if you're on Windows/.NET/Azure.
OpenAIToolsHub
• LangGraph — best for complex stateful workflows, multi-step tool sequencing, and full step-level observability; leads in adoption.
Services Ground
• OpenAI Agents SDK (March 2025) and Claude Agent SDK — leaner, model-family-optimized. The Claude Agent SDK is the simplest path if you're building on Claude.
• Google ADK, CrewAI, Pydantic AI, smolagents round out the field for various role-based or lightweight needs.
These all now treat MCP as the tool-connection substrate, so they're complements to MCP, not alternatives.
One emerging standard to keep on your radar: NLWeb, the open web protocol Microsoft introduced at Build 2025 to connect agents across sites — earlier-stage than the above, but worth watching if your assistant ever reaches out to web services.
