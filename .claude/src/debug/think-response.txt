Link consistency at scale breaks down into two fundamentally different problems that these systems solve in opposite ways: internal links (page-to-page within the system) and external links (out to the open web). The first is solvable with strong consistency guarantees; the second is fundamentally a best-effort, eventual-cleanup problem.
Internal links: the indexed-reference approach
The core insight in both MediaWiki and Confluence is to not store links as raw URLs. Instead, links are parsed at save time and recorded in a separate index that maps source page → target, so the system always knows the full backlink graph without crawling.
In MediaWiki this is the pagelinks table. Each entry contains the source page's ID and a reference to the linktarget table which has the title and namespace that is being linked to from within that source page. A key design detail: links target a namespace + title key, not a page ID. The target page may or may not exist, and due to renames and deletions may refer to different page records as time goes by. This is why a redlink (link to a not-yet-created page) and a working link are stored identically — existence is resolved at render time by joining against the page table, not baked into the link.

That same index powers the "What links here" backlink feature. The pages linking to and/or embedding a given page are listed, and this info comes from the pagelinks table and the templatelinks table. The interesting architectural decision in newer MediaWiki is the linktarget table — an indirection layer where a page namespace-title tuple gets a unique integer ID, used for referenced page titles in templatelinks, pagelinks, and categorylinks. This means a single title is stored once and referenced by integer everywhere, which matters enormously when one popular template is transcluded across millions of pages.

The reason links are stored by title-key rather than page-ID drives how moves work. There are two competing strategies:
Strategy A — Title-keyed links + redirects (MediaWiki). When a page moves, its links don't update at all. The old title becomes a redirect record pointing to the new title, and the link resolution simply follows it. This is cheap (one redirect row, no rewrite of millions of referencing pages) but introduces redirect chains. MediaWiki has a dedicated background job and a maintenance report for double redirects (A→B→C), which a bot flattens to A→C. The tradeoff: links technically still "work" but accumulate a layer of indirection, and "What links here" must account for links arriving via redirect.
Strategy B — ID-keyed links + link refactoring (Confluence). Confluence links resolve through internal page/content IDs rather than titles, so renaming a page leaves all inbound links intact automatically — the title is just a display label resolved from the ID. When a page is moved (reparented or moved across spaces), Confluence runs "link refactoring" that rewrites the stored references so they continue to resolve. The cost is that the move operation is heavier (it touches the referencing content), but there's no redirect-chain accumulation and no ambiguity about whether a link is live.
Both approaches share the same guarantee that web-style links lack: because the system controls both ends and indexes them at write time, an internal link can never silently rot. The worst case is a redlink (clearly flagged) or a redirect (transparently followed).
Scaling the index maintenance
The index has to stay correct as pages change, and at Wikipedia's scale you can't recompute it synchronously. The mechanism is a deferred-update / job-queue model: when you save a page, the parse extracts its links and queues jobs to update pagelinks, templatelinks, etc. The genuinely hard case is template/transclusion changes — editing one widely-used template can invalidate the link sets of millions of pages. MediaWiki handles this with refreshLinks jobs fanned out through the job queue (backed by Kafka/Redis at Wikimedia), processing the backlink set in batches rather than blocking the edit. There are also reconciliation maintenance scripts (refreshLinks.php, pruneUnusedLinkTargetRows.php) that sweep for drift between the rendered content and the link tables, since async updates can occasionally be dropped or left stale.
External links: detection and best-effort repair
This is where you lose all guarantees, because the target is outside your control. The model shifts from "prevent inconsistency" to "detect and remediate."
Indexing still helps for finding references. MediaWiki records outbound external links in the externallinks table, which powers Special:LinkSearch — letting you query, e.g., every page citing a given domain, with reverse-domain indexing (org.wikipedia.en) so you can search by TLD or domain prefix.
Detection is done by crawler bots, not at write time. Wikipedia's InternetArchiveBotmonitors every Wikimedia wiki for new outgoing links and actively makes fixes on over 400 Wikimedia wikis. Its detection logic is more careful than a naive 404 check — it checks whether untagged dead links are actually dead, with a reported false positive rate of 0.1%, and it parses how each URL is used so it can detect if it's in a cite template, a reference, or a bare link, handling sources formatted in various ways. The false-positive discipline matters at scale: even a 1% error rate across millions of links produces tens of thousands of bad edits.

The general pattern
Abstracting across both systems, the toolkit for link consistency at scale is:
1. Parse and index links at write time into a separate structure keyed by something stable — a title-key or, better, an opaque internal ID. Never rely on crawling your own content to know your link graph.
2. Indirection for identity — store a layer (linktarget IDs, content IDs) between "what's linked" and "where it lives," so renames/moves are cheap and don't require rewriting referencing pages.
3. Choose your move strategy: redirects (cheap writes, accumulate chains, need double-redirect cleanup) vs. link refactoring (heavier moves, clean graph).
4. Async fan-out for cascade updates via a job queue, since transclusion and template edits can invalidate millions of pages at once, plus periodic reconciliation sweeps to catch dropped updates.
5. For external links, invert the strategy to detect-and-repair: pre-emptive archiving, crawler-based dead-link detection tuned hard for low false positives, archive substitution, and human-reviewed backlogs.
The unifying principle is that internal consistency is an invariant you maintain (the link index is the source of truth, kept correct by write-time parsing and async jobs), whereas external consistency is an entropy you continuously fight (you can only detect rot after the fact and patch from archives). The systems that scale are the ones that never let the first problem degrade into the second — they keep internal links in a controlled namespace where the system owns both ends.
