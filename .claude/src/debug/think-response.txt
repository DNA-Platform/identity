Synthesized markdown-native tools and caching strategies for wiki navigation
Good — I have a clear picture. One framing point up front, because it changes which tools actually help you: almost all the "make LLM agents better at large file trees" tooling was built for code, where the structure is functions/classes/imports parsed by tree-sitter or an LSP. A prose markdown wiki has different structural primitives — headings, frontmatter, and wikilinks — so the code-native tools give you partial benefit, and the markdown-native ones fit your case (covers, chapters, cross-references, synopsis hierarchy) much better. I'll flag which is which under each of your five problems.
1. Efficient reading and a persistent indexed view
There are two distinct things people conflate here: within-session cheap re-reads and cross-session persistent structure.
For within-session, the real mechanism is Anthropic prompt caching — once a file is in a cached prefix, re-referencing it in the same session is near-free. But it's ephemeral (short TTL) and doesn't survive across sessions, so it's not the answer to "I keep re-reading the same 20 files day after day."
For cross-session persistence, the pattern that works is don't re-read files, query an index of them. The leading examples:
• Aider's repo map builds a structural overview using tree-sitter, ranks files with a PageRank-style graph over symbol references, and renders only the top items into a ~1k-token budget. It's SQLite-cached to avoid re-parsing unchanged files, and the map is regenerated each turn based on what's in the chat. The concept is exactly what you want; the limitation is it extracts code symbols, so on prose markdown it mostly sees headings.
Tekai
• Serena (oraios/serena) is an MCP server that wraps a Language Server and exposes get_symbols_overview, find_symbol, and find_referencing_symbols, so the agent reads a symbol map instead of whole files. Instead of treating your codebase as a pile of text files to grep through, it integrates with an LSP backend and exposes those capabilities as MCP tools, which is meaningfully more token-efficient. Again, code-oriented — useful if your wiki ever embeds code, less so for pure prose.
Arda
• For the markdown case, the equivalent is an MCP server backed by a local SQLite index over your files, which is what basic-memory and the better Obsidian MCP servers do (see #3).
2. Freshness / knowing when to reread
This is a solved problem in practice, with three standard mechanisms, increasing in robustness:
• mtime + content hashing. Both Aider and basic-memory key their caches on file modification and only re-process what changed. This is the cheapest reliable approach: keep a manifest of (path, content_hash, synopsis), and on session start diff the hashes to find which synopses are stale.
• Filesystem watchers. drewburchfield's obsidian-graph ships a file_watchercomponent that re-indexes on change in real time, and basic-memory does bidirectional sync between the files and its index.
• Git-aware. git status / git diff against the last-indexed commit tells you exactly what moved since the previous session — and both repomix and code2prompt can fold git diffs into context so the agent sees what changed rather than re-reading everything.
For your setup the content-hash manifest is the sweet spot: deterministic, no daemon, and it directly answers "is this cached synopsis still valid."
3. Database / vector store / knowledge graph
For ~200 files, resist the urge to reach for a vector DB first. The hierarchy of options:
• SQLite FTS is the right default. basic-memory stores content in Markdown and uses a SQLite database for search indexing, extracting Observations and Relations to build a knowledge graph. Keyword + frontmatter search over 200 files is instant and preserves your link structure if you also store an edges table.
Aibase
• The link graph itself is your knowledge graph. Your bidirectional links are the edges — you don't need to synthesize a graph, you need to make it queryable. The good Obsidian MCP servers expose exactly this: graph-neighbors does a breadth-first search from any note, finding every connected note within a configurable depth, with direction control for outgoing links, incoming backlinks, or both, plus orphan/broken-link detection. That lets the agent traverse "from this cover, depth 2" as structured data instead of reading and following links one file at a time.
Mcpbundles
• Vector store (Chroma, LanceDB, or pgvector as in obsidian-graph) earns its place only for semantic "find conceptually related notes I didn't explicitly link." It adds embedding cost and its own staleness problem. Notably, the Karpathy-style LLM-wiki community has found that at roughly 400K words, context windows handle related clusters efficiently without heavy vector RAG — at your scale, FTS + link-graph traversal will usually outperform a vector setup on effort-to-value.
a2a mcp
4. Repos that optimize LLM file operations
The most useful ones, by category:
• Token-aware packing: repomix (packs the tree into one AI-friendly file, per-file token counts, and repomix --token-count-tree 1000 to show only directories above a token threshold); code2prompt (Rust, Handlebars templating, token counting, git diffs, MCP server mode); simonw's files-to-prompt; and gitingest. These are how you'd build the "cover + chapter pack" for a given exploration without manually stitching files.
• Smarter search: ripgrep for raw speed, ast-grep for structural/heading-aware search. For markdown, a heading-aware extractor (pull just the section under a matching ##) beats whole-file grep.
• Semantic navigation MCP: Serena as above.
• Markdown-native MCP servers: basic-memory and the feature-complete Obsidian servers (one advertises 23 tools for search, link analysis, graph traversal). One basic-memory build even uses FastMCP with tool annotations (readOnlyHint, destructiveHint, idempotentHint) so agents discover capabilities progressively instead of burning tokens guessing.

5. Practical recommendation for your specific wiki
Your four-layer synopsis hierarchy is the most valuable thing you have, and it's worth recognizing what it is: a hand-built, prose version of Aider's auto-generated repo map. So don't fight it — make it the spine of the system. Concretely:
Lean on progressive disclosure. Make the top synopsis layer (plus an INDEX.md manifest) the mandatory entry point. The agent reads the top layer, then descends a level only for the branch it actually needs, and only reaches full chapter text at the leaf. This is precisely the pattern that lets Karpathy-style wikis stay tractable: the LLM navigates the full corpus leveraging self-maintained indexes and summaries rather than loading raw files. With four layers you can answer most queries from layers 1–2 and never touch most chapters.
a2a mcp
Add a thin index, not a heavy one. A small SQLite database with three tables — files(path, hash, synopsis_L1..L4), links(src, dst), and an FTS table over content — gives you full-text search, instant backlink lookup, and the freshness manifest in one place. You can get this turnkey from basic-memory (its entities/observations/relations model maps cleanly onto your covers/chapters/cross-refs, and it already does the SQLite indexing and bidirectional sync), or via an Obsidian MCP server if you keep the vault in Obsidian, or as a ~150-line custom MCP server — a markdown-wiki indexer is genuinely one of the simplest MCP servers to write, and rolling your own means the synopsis-hierarchy logic is exactly yours rather than bent to fit someone else's schema.
Freshness via hash manifest. On session start, hash every file, diff against the stored manifest, regenerate synopses only for changed files. Cheap, deterministic, no watcher daemon.
Avoid the context burn directly. The reason you refill the window is the agent re-reads files it has effectively already "known" via summaries. Two fixes: (a) the agent queries the index/synopsis layer and only opens full files at the moment of need, and (b) keep an explicit read-log in the session so it doesn't reopen what it has open. That second discipline — an explicit working set — is the same one you already apply to the SRT theory work, and it's the single highest-leverage habit here.
If I were standing up a v1 today for 200 files: keep the synopsis hierarchy as-is, point basic-memory (or a small custom MCP server) at the vault for SQLite FTS + link-graph queries, add a hash-manifest freshness check, and skip embeddings until you hit a concrete "semantic find" need that FTS can't serve. Add Serena only if code starts living in the wiki.
