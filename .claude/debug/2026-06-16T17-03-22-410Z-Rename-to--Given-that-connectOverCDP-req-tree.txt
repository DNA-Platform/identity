ControlType.Text | Short version: the port really is gated at process initialization, and nothing in stock Chromium/Electron lets you turn the browser-level CDP server on after the fact. But the two protocols you're conflating behave differently, and one of them gives you a live-process path. Let me separate them.
ControlType.Text | The browser CDP endpoint (what
ControlType.Text | connectOverCDP
ControlType.Text |  needs)
ControlType.Text | Playwright's
ControlType.Text | connectOverCDP
ControlType.Text |  talks to the Chromium
ControlType.Text | DevToolsHttpHandler
ControlType.Text |  — the HTTP server at
ControlType.Text | http://host:port/json/version
ControlType.Text |  that hands back the browser
ControlType.Text | webSocketDebuggerUrl
ControlType.Text | . That handler is constructed during browser-process startup
ControlType.Text | if and only if
ControlType.Text |
ControlType.Text | --remote-debugging-port
ControlType.Text |  (or
ControlType.Text | --remote-debugging-pipe
ControlType.Text | ) is present in the command line. There is no runtime API — no
ControlType.Text | app.enableRemoteDebugging()
ControlType.Text | , no IPC, no documented switch — to start it later. So an untouched, already-running process simply has no such server, and there's no clean way to make one appear.
ControlType.Text | Environment variables don't help here. Chromium has no general
ControlType.Text | CHROME_REMOTE_DEBUGGING_PORT
ControlType.Text | -style env var that injects the switch; the only env vars Electron honors broadly are things like
ControlType.Text | ELECTRON_RUN_AS_NODE
ControlType.Text | , which is irrelevant to this. Unless the app's
ControlType.Text | own
ControlType.Text |  code reads an env var and forwards it as a flag (some do — worth grepping the app's main bundle), env alone is a dead end.
ControlType.Text | The realistic no-source-change method: relaunch with package identity
ControlType.Text | The thing that actually works for an MSIX app without touching its source is to relaunch it with the flag while
ControlType.Text | preserving package identity
ControlType.Text | . You can't just exec the inner exe out of

ControlType.Text |  — it'll usually run without identity and break storage/state. And shell activation (
ControlType.Text | shell:AppsFolder\<PFN>!<AppId>
ControlType.Text | ) doesn't reliably forward arbitrary args. The documented tool is the

ControlType.Text |  module cmdlet:
