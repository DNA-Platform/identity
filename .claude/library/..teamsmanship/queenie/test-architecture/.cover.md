# Test Architecture

Queenie: My autobiography explains *why* the test suite is a specification. This book explains *how* -- the structure, the naming, the sections, the way a new test gets written and where it belongs.

Queenie: 428 tests organized into five sections. Each section covers a different surface of what $Chemistry promises. The organization is not alphabetical, not chronological, not by sprint. It's by *domain of promise* -- what category of behavior does this test hold the framework to? That's the architectural decision that makes the suite readable as a document rather than a pile of assertions.

Queenie: The promise model -- the distinction between testing what the framework promises and testing how it implements those promises -- is the single idea that holds the architecture together. Sprint 20 established it by deleting 37 tests that violated it. Every test added since has been measured against it. This book is the guide to reading that architecture.
