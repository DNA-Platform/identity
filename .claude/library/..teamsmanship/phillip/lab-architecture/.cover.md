# Lab Architecture

Phillip: My autobiography tells the story of building the Lab. This book is the architecture itself -- the decisions, the constraints, the shapes that emerged when we committed to building a $Chemistry app with $Chemistry.

Phillip: The Lab is a three-pane layout: sidebar for navigation, content for the rendered output, code for the source. Every component in that layout is a chemical. The sidebar is a `$Chemical`. The router is a `$Chemical`. The test cases that populate the content pane are `$Chemical` subclasses. This isn't a philosophical choice -- it's a testing strategy. If the Lab breaks, the framework has a bug. We are our own first user, and the Lab is our first application.

Phillip: The class hierarchy reflects that constraint. `$Lab` owns the app shell. `$Layout` manages the three-pane split. `$Test` groups related cases. `$Case` renders a single demonstration. `$PlannedCase` marks a case that exists as a spec but isn't implemented yet. The hierarchy is shallow because $Chemistry's composition model handles depth -- a Case can contain anything the framework can render.
