# Sprint 64: The Audit Skill and Compilation

- **author:** [Arthur](../..teamsmanship/..team/arthur/arthur-or-the-shape-of-everything/.cover.md)

---

Build a centralized compilation catalogue and the `/audit` skill that exercises it. Compilation isn't just environment-facing output — it's every automated process the library performs: validation, compilation, sync, and whatever we automate next. The catalogue specifies what automation exists; the audit skill runs all of it.

## Why this sprint

Sprint 63 proved the branch library works at scale: 9 books, 58 chapters, 402 links, all valid. But validation required running multiple tools manually, and there's no single command that checks everything. The commit tool runs bookkeeping + compiled-links; the full link checker runs separately; the branch audit was done ad hoc. We need one command that means "check everything," and a specification that says what "everything" means.

## Task 1: Create the Compilation subject catalogue — Libby

Create `.compilation/` as a subject catalogue in the identity library with `catalogues: Composition`. Catalogued by Librarianship.

Compilation catalogues every automated process the library performs — not just environment-facing compilation, but all automation:

- **Compilers:** teammate files (01-on-teammates--compiler.ts), CLAUDE.md (02-on-bootstrap--compiler.ts), rules (03-on-rules--compiler.ts), skills (04-on-skills--compiler.ts)
- **Validators:** bookkeeping (11-on-specifications--validator.ts), compiled links (07-on-compiled-links--validator.ts), link checker (05-on-validation--check-links.ts)
- **Sync tools:** the commit tool (06-on-sync--commit.sh), branch routing, robocopy sync
- **The runner:** validation pipeline (05-on-validation--runner.ts)
- **The constraint:** every autogenerated file must have provenance pointing to its generator
- **Future:** book-to-book sync, cross-branch validation, whatever automation the library grows into

Compilation explains WHY audit reads it: auditing the library IS auditing its composition. The catalogue is the specification; the audit skill is the automation.

Chapters:
1. What Compilation catalogues — the full inventory of automated processes, not just compilers
2. The provenance constraint — every compiled file links to its compiler
3. The audit loop — why reading Compilation is part of auditing (the audit checks the thing that specifies the audit)

## Task 2: Build the `/audit` skill — Claude, Adam

Adam owns the runner infrastructure — the central script that orchestrates every check. Claude owns the environment-facing validators and compilers. Together they build the skill.

The audit should:

1. Run the bookkeeping validator against the identity library
2. Run the bookkeeping validator against all branch libraries (`.lib/` directories)
3. Run the compiled-links validator
4. Run the full link checker against the identity library
5. Run the full link checker against all branch libraries
6. Verify all compilers produce output matching current files (recompile + diff)
7. Verify branch separation (`.lib/` only on project branches, not on `main` or `dna-platform`)
8. Report: books checked, chapters checked, links checked, broken links, stale compiled files

Adam builds the runner that wires these checks together. Claude specifies each check and builds the compiler-staleness verification.

The skill must be autogenerated and must have provenance pointing to its generator.

## What success looks like

- `.compilation/` exists with a cover and 3 chapters
- `/audit` skill exists and passes when run
- Running `/audit` validates the full library (identity + branches) with one command
- Every autogenerated file has provenance, verified by audit
- Librarianship catalogues Composition
- Branch separation is verified automatically, not ad hoc
