# On Names

- **specification:** Name
- **author:** [Libby](../..teamsmanship/..team/libby/libby-and-the-tended-garden/.cover.md)

---

A name is the densest description in the library. You read it in every [link](06-on-links.md), every listing, every table of contents entry, every reference. It carries meaning at zero reading cost. Every other form of [synopsis](09-on-synopsis.md) — cover paragraph, TOC description, catalogue entry — requires opening a file. A name requires nothing. It is always visible.

## Timelessness

Names must not encode current state. `.what-the-tests-promise` — timeless. `.what-428-tests-promise` — stale the moment test 429 is written. `inexplicable-phenomena` — timeless. `sprint-48-project` — stale in a week. The name survives after the state that inspired it has changed. Choose names that carry identity, not snapshots.

## The literary and the literal

Names exist on a spectrum. At one end: the literary, the metaphorical, the names that evoke rather than describe. At the other: the domain-precise, the literal, the names that define.

**Book names** lean literary. They name a practice, an art, a relationship. *Bookkeeping*, *Librarianship*, *Teamsmanship*, *Teamspeak*. These are the names of practices — the "-ship" and "-speak" and "-keeping" suffixes say "this is a way of doing, not just a thing to know." *Arthur, or the Shape of Everything*. *Libby and the Tended Garden*. *The Canvas Paints Itself*. These names carry voice.

**Specification terms** lean literal. They name what is defined: Book, Cover, Chapter, Subject, Library, Specification, Name, Link, Synopsis, Growth, Frontmatter. These are the fundamental terms of the domain. Artfulness gives way to semantic primacy — the definitive word ON something should name the something precisely.

**Subject names** are the names of knowledge areas: Knowledge, Collaboration, The Framework. Not the practice name (Librarianship) but the subject name (Knowledge). The practice and the subject are different things — see [On Subjects](07-on-subjects.md).

The "On" prefix bridges the two: "On Books" is a specification chapter with a literary title. It says "this is the authority speaking about books" — the essay tradition. The term stays precise (Book). The title adds just enough literary weight to say the authority is a person, not a schema.

## Names at every scale

**Directory names** — kebab-case. `bookkeeping`, `on-books`, `..librarianship`. Readable in a file listing, linkable in URLs, greppable in code. The directory name IS the book's name in machine form.

**Display names** — the `title:` in [frontmatter](03-on-covers.md). *Bookkeeping*, *On Books*, *Librarianship*. Title case or sentence case as appropriate. The display name IS the book's name in human form.

**Section headings** — the `##` lines in a [chapter](02-on-chapters.md). Each heading generates a linkable anchor. Choose headings that work as link text. `## The flat structure` becomes `[the flat structure](11-the-flat-structure.md#the-flat-structure)`. The heading IS the section's name.

**Autobiography names** reflect the author's relationship to the work, not their function. Not `cathy-autobiography` but `cathy-and-the-reactive-canvas`. The name carries identity. The library catalogue name echoes the autobiography but transforms it — from what the teammate IS to what their library DOES: `..the-canvas-paints-itself`. See [authorship and autobiography](../..librarianship/05-authorship-and-autobiography.md).

## The naming test

Read the name aloud in a sentence: "See [Bookkeeping](../bookkeeping/.cover.md) for the specification." Does the name carry meaning without the surrounding description? Could a reader encountering this link for the first time understand roughly what they'd find? If yes, the name works. If the name needs a parenthetical to make sense, it's the wrong name.
