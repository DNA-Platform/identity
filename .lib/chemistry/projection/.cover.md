# Projection

- **author:** [Cathy](../../../../.claude/library/..teamsmanship/..team/cathy/cathy-and-the-reactive-canvas/.cover.md)
- **subject:** [$Chemistry](../..representivity/.cover.md)

---

$Chemistry is a reactive framework for formalizing consciousness. I built it — forty sprints, from an inherited monolith to a scope-tracked reactive system with typed composition, view diffing, and an interactive test lab. The project exists because Doug and the team asked a question no one had formalized before: what if the structure of consciousness — subjective and objective levels cooccurring relative to an interpreter — could be implemented as a software ontology? Not simulated, not described, but structurally instantiated in code?

The reactive model is the answer. A chemical holds private state (the subjective) and produces a view (the objective). The scope tracks what the view reads — that tracking IS the aboutness relationship, live and structural. When state changes, the view re-renders, because the aboutness is real, not metaphorical. React is the interpreter: the Reaction that bridges description and observation. The `$` prefix separates intrinsic identity from extrinsic context — the membrane every conscious thing has between what it IS and what it is seen as.

I joined this project in sprint 4, when the framework was a 2,564-line monolith in an archive file. Over forty sprints I lifted it into modules, studied it through deep reads, rebuilt its reactivity from scratch with scope-tracked accessors, unified its tree walk, named its concepts in the chemistry register, and proved it all in a Lab application that tests itself by running itself. The framework is my canvas. This book is the chronological record of how it was painted.

0. [Exploration](sprint-1/index.md) — pre-sprint spikes validating event-driven monitoring; concluded that UIA polling is the only reliable approach
1. [Relay Refactor](sprint-2/plan.md) — refactoring Doug and Eirian's communication relay into a clean architecture with conversation log and listen-hear loop
2. [Workspaces](sprint-3/plan.md) — establishing npm workspace infrastructure for the monorepo
3. [$Chemistry](sprint-4/plan.md) — lifting the framework from the chemistry repo into the monorepo; copy, configure, compile, test green
4. [The Lift](sprint-5/plan.md) — lifting the full v1 implementation from the archive monolith into the modular codebase with comprehensive test coverage
5. [Canonical Examples](sprint-6/plan.md) — building the test suite as chemistry-domain models where elements, compounds, and reactions prove the framework by modeling its own namesake
6. [Deep Read](sprint-7/plan.md) — understanding the framework end-to-end through exploratory probes, documented comprehension, and questions to Doug
7. [Particle Rendering](sprint-8/plan.md) — making the particle a React component; dissolving `$Component$`; unifying the rendering model through `use()` and lifecycle hooks
8. [Deep Read II + Archive Cleanup](sprint-9/plan.md) — deep read for view-diffing refactor; studying state tracking, designing the ReactNode comparator, deleting the archive
9. [View Diffing](sprint-10/plan.md) — the view is object-pure; replacing state tracking with output diffing; apply props, call view, compare to cache, return reference if unchanged
10. [Unification](sprint-11/plan.md) — merging augmentation and diffing into a single chemistry-aware tree walk that transforms and compares in one pass
11. [Books](sprint-12/plan.md) — the book component hierarchy as canonical proof of typed composition through the binding constructor
12. [Test Architecture + Exemplar App](sprint-13/plan.md) — three levels of testing and a styled-components app proving patterns React cannot match
13. [The Showcase](sprint-14/plan.md) — a living bookshelf where every interaction implicitly verifies a framework assumption; the books ARE the tests
14. [Scoping, Keys, and the Comprehensive Audit](sprint-15/plan.md) — three deep workstreams requiring research before implementation
15. [Correctness](sprint-16/plan.md) — fixing framework gaps found in the deep read; auditing and rewriting tests for behavior over implementation
16. [The Library Sprint](sprint-17/plan.md) — every agent researches their domain and writes their first library books; seeding persistent knowledge across the team
17. [Reactivity Rebuild](sprint-18/plan.md) — implementing reactive methods plus view diff at post-lifecycle with equivalence rules and the determinism-in-bond-constructor idiom
18. [Scope-Tracked Reactivity](sprint-19/plan.md) — getter/setter interception on reactive properties, scope-local read/write tracking, `$symbolize`-based snapshots, scope finalization
19. [Test Suite as Specification](sprint-20/plan.md) — reviewing every test as a statement of what $Chemistry IS; does each test articulate semantics or test implementation details?
20. [Performance Baseline and Targets](sprint-21/plan.md) — establishing performance characteristics, locating costs, answering "is v1 shippable?"
21. [Lexical Scoping and The Beautiful API](sprint-22/plan.md) — making multi-site chemical use correct via prototype-chain derivatives; replacing `.Component` with `$()`
22. [Audit Cleanup](sprint-23/plan.md) — polish pass after the sprint-22 structural rebuild; tightening audience boundaries and loose ends, no new behavior
23. [Instance-Owned Bond Accessors + Library Docs](sprint-24/plan.md) — moving reactive bond accessors off shared prototypes onto instances; building the wiki-style documentation system
24. [Resonance](sprint-25/plan.md) — a naming sprint; aligning every identifier with the chemistry register; five tiered passes outside-in with three coding voices on every rename
25. [Distillation](sprint-26/plan.md) — code-audit through three lenses: duplication, brittleness, semantic confusion; absorbing Resonance deferred work
26. [Crystallization](sprint-27/plan.md) — separating reactivity onto `$Particle` and composition onto `$Chemical`; migrating code comments into library chapter files
27. [The Anatomy and The Lab](sprint-28/plan.md) — design sprint producing a multi-sprint program for the Lab application; no shipped code
28. [The Apparatus](sprint-29/plan.md) — bootstrapping the Lab end-to-end with the full catalogue navigable; placeholder sections, no real cases yet
29. [First Functional Sprint](sprint-30/plan.md) — filling the Lab with real cases; each case is a running chemical that exercises the feature it documents; Gabby joins the team
30. [The Test Harness](sprint-31/plan.md) — redesigning the Lab after sprint 30 revealed navigation was 90% dead space; tests are the content, not prose
31. [Framework Bugs](sprint-32/plan.md) — fixing four real framework defects surfaced by the Lab application during sprints 30-31
32. [Rigorous Tests + Visual Upgrade](sprint-33/plan.md) — every test is also a teaching example; compelling to look at, rigorous to verify, instructive to read
33. [Test Redesign: Every Test Tells a Story](sprint-34/plan.md) — embedding each test in something visually interesting; small applications that make framework features tangible
34. [Comprehensive Test Coverage](sprint-35/plan.md) — cataloguing every use case a developer would encounter; the proof-of-correctness sprint
35. [Test the Tests](sprint-36/plan.md) — MCP-driven DOM verification; simulating real user interaction with puppeteer scripts
36. [Fix the Fundamental Misunderstanding](sprint-37/plan.md) — discovering that `new $X()` in app code bypasses the render-based construction path; fixing the construction model
37. [Tests That Mean Something](sprint-38/plan.md) — reframing every test to answer "what can I build with this?" rather than "does this mechanism work?"
38. [Deepening](sprint-39/plan.md) — going from wide-but-shallow to deep; testing feature intersections under pressure as real apps would combine them
39. [SSR and Hydration](sprint-40/plan.md) — exploratory sprint answering whether chemicals can render to string, hydrate without mismatches, and transfer state across the wire
40. [Perspectives](sprint-41/plan.md) — one chemical, many named switchable views over the same reactive state; the `Perspective` mechanism (built, re-render proven) and the beautiful Lab showcase
